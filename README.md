# 🔐 AMessenger — Защищенный корпоративный мессенджер

**AMessenger** — это Android-клиент для защищенного корпоративного мессенджера с End-to-End шифрованием, интегрированный с системой контроля версий (GitLab) и HR-системой (Odoo).

Проект разработан в одиночку (включая проектирование, клиентскую часть и Backend) и успешно защищен в качестве выпускной квалификационной работы (ВКР бакалавра). Приложение обеспечивает высокий уровень безопасности, бесперебойную работу в реальном времени, поддержку оффлайн-режима и продвинутую работу с медиафайлами.

## 🚀 Ключевой функционал

* **Полноценный опыт мессенджера:** Реализованы все базовые и необходимые функции: ответы на сообщения, пересылка (forward), запись голосовых сообщений. Также поддерживается удаление чужих сообщений в чате (если данная опция включена в настройках конкретного чата).
* **Автоудаление сообщений (Auto-deletion):** Возможность включить в настройках таймер автоудаления сообщений (например, 30 секунд или 5 минут). Очистка гарантированно обрабатывается на стороне сервера с использованием очереди задач на базе **Redis + Dramatiq**.
* **Инструменты для разработчиков:** Встроенный редактор кода с подсветкой синтаксиса (на базе `CodeView`), поддержка кликабельных именованных ссылок, получение push-уведомлений о событиях в репозиториях GitLab (push, merge, issue).
* **HR-интеграция (Odoo):** Автоматическая блокировка доступа к интерфейсу приложения на время отпуска сотрудника (поверх UI выводится блокирующий фрагмент), выдача ролей (например, "редактор новостей") через HR-систему.
* **Новостная лента:** Отдельный раздел корпоративных новостей с поддержкой медиа-вложений, кэшированием и пагинацией (реализовано через `Jetpack Paging 3`).
* **Офлайн-режим и работа с памятью:** Локальное кэширование сообщений, новостей и медиафайлов. Приложение полностью функционально для чтения без интернета. Встроен умный механизм очистки: при обновлении БД приложение автоматически удаляет "осиротевшие" файлы старых сообщений, предотвращая засорение памяти устройства.
* **Кастомизация UI:** Светлая и темная темы (по 3 варианта цветовых схем для каждой), выбор фоновых обоев для чатов (с адаптивным разделением на темные и светлые).

## 🛡 Безопасность и Криптография (Security)

Безопасность — ключевой фокус проекта. Использована связка криптографических библиотек для реализации надежного сквозного (E2E) шифрования:

* **End-to-End шифрование (Google Tink):** Полное шифрование текстов сообщений и прикрепляемых файлов с использованием симметричных алгоритмов (AES-256 / AEAD). Для каждого чата генерируются свои уникальные ключи.
* **Механизм Master-Key (Java Security):** Инновационный подход для E2E-мессенджеров, позволяющий восстановить доступ к переписке при смене устройства или переустановке приложения. Асимметричная пара ключей (RSA) генерируется на клиенте. Приватный ключ пользователя шифруется с помощью его личного Master-Key (пароля, который знает только он) и в таком виде отправляется на сервер. При входе с нового устройства пользователь вводит Master-Key, расшифровывает свой приватный ключ и возвращает себе доступ ко всем зашифрованным чатам.
* **Android Keystore & Bouncy Castle:** Безопасное аппаратное хранение всех криптографических материалов на устройстве. Использование Bouncy Castle для работы с сертификатами.

## 🛠 Технический стек и Архитектура

Приложение написано на **Kotlin** с использованием XML-верстки (ConstraintLayout + кастомные ViewGroup) и базируется на архитектурном паттерне **MVVM**.

* **Сетевое взаимодействие и Авторизация (`Retrofit 2` + `OkHttp3` + `Moshi`):** Написан кастомный Interceptor, который автоматически извлекает JWT-токен из SharedPreferences и "зашивает" его в Headers каждого запроса. Если токен отсутствует или сервер возвращает ошибку неавторизованного доступа, Interceptor мгновенно прерывает сессию и перебрасывает пользователя на экран авторизации.
* **Real-time и Уведомления:** Основной канал связи — WebSocket (`Socket.IO`) для мгновенной доставки сообщений в активном приложении. Вспомогательный канал — Firebase Cloud Messaging (`FCM`), который используется для "пробуждения" соединения и доставки внешних уведомлений (например, от GitLab).
* **Асинхронность и потоки данных:** Coroutines, StateFlow, SharedFlow, LiveData.
* **Локальная БД:** Room (хранение истории, диалогов, файлов).
* **Медиа и UI:** * `Glide` — кастомный механизм двухэтапной загрузки картинок для экрана вложений (сначала грузятся минимальные preview, а при открытии — "на лету" подгружаются и заменяются оригиналы для экономии RAM).
  * `PictureSelector` + `Luban` — выбор медиа с мощным алгоритмом сжатия "на лету" перед отправкой.
  * `ExoPlayer` — воспроизведение видео и аудио.
* **DI:** Dagger Hilt.

## 🧠 Инженерные решения и Оптимизация

* **Клиентский поиск (Inverted Index):** Из-за E2E-шифрования сервер не имеет доступа к тексту сообщений, поэтому классический серверный поиск невозможен. Проблема решена на клиенте: приложение запрашивает зашифрованные сообщения, локально обрабатывает их и использует методику инвертированного индекса (Inverted Index) для реализации быстрого поиска по ключевым словам.
* **Глубокая переработка работы с Audio (Voice messages):**
  * **Конвертация PCM -> AAC "на лету":** Из-за устаревания `FFmpeg` (который использовался ранее для конвертации PCM в OGG), был написан собственный движок на базе `AudioRecord`, `MediaCodec` и `MediaMuxer`. Теперь звук пишется в сырой PCM и "на лету" конвертируется в AAC (`.m4a`). Параллельно высчитывается и записывается массив данных для красивой отрисовки осциллограммы (waveform), так как UI-библиотека не умеет напрямую работать с сырым PCM.
  * **Реанимация Legacy UI-компонента:** Нужная библиотека с готовым UI-интерфейсом для диктофона (`audiorecorderview`) была удалена автором из JitPack. Исходный код UI-компонента был вручную перенесен в проект. Его интерфейсы (`onRecordStart`, `onRecordEnd`) были переопределены во фрагменте и "подружены" с новым кастомным VoiceRecorder-ом.
* **Кастомная Cursor-пагинация чатов:** Из-за конфликтов стандартного `Jetpack Paging 3` со сложным стейт-менеджментом чата, пагинация сообщений была написана с нуля на базе `ListAdapter`. В связке с сервером реализован механизм Cursor Pagination (по временной метке). Это гарантирует бесшовную подгрузку истории без скачков UI и дублирования сообщений в моменты, когда в чат прилетают новые данные в реальном времени. (Стандартный Paging 3 оставлен только для простых списков, вроде новостей).
* **Глобальный рефакторинг адаптеров:** Был проведен масштабный рефакторинг: из `RecyclerView` адаптеров полностью убрана бизнес-логика и прямые вызовы `ViewModel`. Теперь все сложные операции происходят во `ViewModel`, а адаптеры получают обновления строго через Sealed-классы (состояния загрузки, ошибки, успех стейтов).
* **DRY и абстракции:** Логика личных и групповых чатов совпадает на 85%. Для соблюдения принципа DRY разработаны базовые абстрактные классы (Abstract Fragments & ViewModels), от которых наследуются конкретные реализации.
* **Управление жизненным циклом:** Надежная обработка конфигурационных изменений (повороты экрана) через `onSaveInstanceState` и восстановление состояния из `Bundle`.

## 🖥 Backend (Кратко)

Серверная часть разработана мной специально под нужды мессенджера:

* **Стек:** Python, Flask, SQLAlchemy, PostgreSQL.
* **Фоновые задачи:** Очередь задач **Redis + Dramatiq** (используется для автоудаления сообщений по таймеру).
* **Особенности:** Партицирование таблиц базы данных (чистый SQL поверх ORM для разделения истории сообщений по разным таблицам для ускорения выборок), JWT-авторизация, обработка Webhooks от GitLab, кастомные add-ons для взаимодействия с Odoo.
